# WARP.md

This file provides guidance to WARP (warp.dev) when working with code in this repository.

## Project Overview

This repository is a three-phase Infrastructure as Code (IaC) solution for managing Proxmox infrastructure (LXC containers, Linux VMs, Windows VMs, and a K3s cluster):

1. **Terraform (Phase 1)** – Provisions infrastructure and generates unique SSH keys per host.
2. **Ansible (Phase 2)** – Configures hosts (bootstrap user, install Docker/Podman, harden SSH, manage K3s).
3. **GitLab CI/CD (Phase 3)** – Deploys Docker services using a multi-stage pipeline and dynamic child pipelines per service.

Terraform creates infrastructure → Ansible configures hosts one-by-one → GitLab CI/CD deploys services.

## Key Directories & Entry Points

- `terraform/` – Three independent Terraform projects:
  - `terraform/lxc/` – LXC containers.
  - `terraform/linux-vms/` – Linux VMs (Debian/Ubuntu with cloud-init).
  - `terraform/windows-vms/` – Windows VMs (Windows Server with cloudbase-init).
- `ansible/` – Unified Ansible project for all Proxmox hosts and the K3s cluster.
- `services/` – Per-service Docker Compose definitions and metadata.
- `scripts/ci/` – Python + Bash CI/CD orchestration for linting and deployments.
- `docker/` – CI and service Dockerfiles (e.g., `docker/ci/`).

Refer to `README.md` in the repo root for an in-depth narrative overview.

## Common Commands & Workflows

### Terraform – Provisioning per Project

Commands are run from the specific project directory (`terraform/lxc`, `terraform/linux-vms`, or `terraform/windows-vms`). Prefer the Makefile targets over raw `terraform` commands.

**Initial setup (any Terraform project):**

```bash
cd terraform/<project>
make setup       # Create terraform.tfvars.secret and init providers
make help        # Discover all project-specific commands
```

**Typical plan/apply cycle:**

```bash
cd terraform/<project>
make plan        # Preview changes (includes instances/*.auto.tfvars)
make apply       # Apply changes and update state
make destroy     # Destroy all resources for this project (with prompt)
make inventory   # Generate project-specific Ansible inventory
make status      # Summarize current deployment state
```

Where `<project>` is one of `lxc`, `linux-vms`, or `windows-vms`.

#### Linux VMs – Extended Makefile Workflow

In `terraform/linux-vms/` there is a comprehensive Makefile with higher-level workflows (see `docs/MAKEFILE.md` for full reference). The most useful targets are:

```bash
cd terraform/linux-vms
make setup         # Secrets + terraform init
make quick         # fmt + validate + plan
make deploy        # init + plan (then interactive apply)
make full-deploy   # init + plan + apply + inventory
make test          # Terraform tests (if using Terraform >= 1.6)
make drift-detect  # Detect configuration drift
make security-scan # Aggregated tfsec/checkov/terrascan (if installed)
```

Use `make import-vm-linux` and `make import-guide` if you need to import existing Proxmox VMs into Terraform state.

### Ansible – Host Configuration & K3s Management

Commands are run from `ansible/`. This project works off inventories generated by the Terraform projects.

**Inventory sync (must be refreshed after Terraform changes):**

```bash
cd ansible
make inventory-all          # Sync and merge LXC, Linux VMs, Windows VMs, K3s
# or:
make inventory-lxc          # LXC only
make inventory-linux-vms    # Linux VMs only
make inventory-windows-vms  # Windows VMs only
make inventory-k3s          # K3s only
```

**Bootstrap & configure Proxmox hosts (single-host philosophy):**

```bash
cd ansible
make accept-host-keys               # Manage SSH known_hosts for all hosts

# One host at a time – bootstrap is ONE-TIME and runs as root
make bootstrap-host HOST=<hostname> # Create non-root user, copy SSH keys, harden SSH

# One host at a time – configure runs as the non-root user (e.g. maintainer)
make configure-host HOST=<hostname> # Install Docker/Podman, create /srv/docker, hardening

# Connectivity checks
make ping                           # Ping all hosts
make ping-host HOST=<hostname>      # Ping a specific host
```

**K3s cluster operations:**

```bash
cd ansible
make k3s-bootstrap       # Initial K3s cluster bootstrap
make k3s-verify          # Health checks
make k3s-update          # Rolling cluster update
make k3s-diagnostics     # Collect diagnostics
make k3s-system-update   # OS updates on K3s nodes

# After bootstrap
export KUBECONFIG=$(pwd)/k3s/kubeconfig/config
kubectl get nodes
```

**Linting and validation for Ansible content:**

```bash
cd ansible
make lint          # ansible-lint
make syntax-check  # Ansible playbook syntax validation
```

### CI/CD Scripts & Tests – `scripts/ci/`

The CI pipeline is driven by Python orchestration and Bash infrastructure scripts in `scripts/ci/`.

**Run CI Python tests locally:**

```bash
cd scripts/ci
pytest -v                         # Run all tests
# Example: run a single test (adjust test name as needed)
pytest test_generate_pipeline.py::test_name
```

**Run linters locally (same logic as CI jobs):**

```bash
# From repo root
python3 scripts/ci/lint_yaml.py --format gitlab      # YAML linting with Code Quality
python3 scripts/ci/lint_markdown.py --verbose        # Markdown linting
python3 scripts/ci/lint_shell.py --verbose           # ShellCheck via Python wrapper
```

**Manual service lifecycle helpers (on your workstation):**

```bash
# From repo root
./scripts/ci/validate-service.sh <service>
./scripts/ci/preflight-check.sh <service>
./scripts/ci/backup-service.sh <service>
./scripts/ci/deploy_service.sh <service>
./scripts/ci/health-check.sh <service>
```

These scripts rely on the same environment variables and GitLab CI/CD variables as the pipeline (`SSH_USER`, `DOCKER_COMPOSE_DIR`, `SERVICES_DIR`, and `SSH_KEY_*` / `HOST_*` / `ENV_*`).

### Services & GitLab CI/CD

Each service lives under `services/<service>/` and is deployed by the GitLab CI/CD pipeline to `/srv/docker/<service>/` on the appropriate host.

**Service directory structure:**

- `services/<service>/docker-compose.yml` – Required Docker Compose definition.
- `services/<service>/.service.yml` – Required metadata, at minimum:
  - `target_host: <inventory-hostname>` – Must match an Ansible inventory hostname.
- `services/<service>/README.md` – Optional service docs.

**Deployment via GitLab CI/CD:**

- Push to `main`:
  - Full pipeline runs; service deploy jobs require **manual approval** on `main`.
- Push a tag (e.g., `v1.0.0`):
  - All changed services auto-deploy without manual approval.
- Merge Requests:
  - Linting and validation jobs run; deployment does not occur.

The CI pipeline (`.gitlab-ci.yml`) uses `scripts/ci/detect_services.py` and `scripts/ci/generate_pipeline.py` to discover changed services and generate a dynamic child pipeline with per-service jobs: `validate → preflight → backup → deploy → verify`.

## High-Level Architecture & Data Flows

### Phase 1: Terraform – Infrastructure Projects

- **Projects:**
  - `terraform/lxc/` – Provisions LXC containers.
  - `terraform/linux-vms/` – Provisions Linux VMs with cloud-init.
  - `terraform/windows-vms/` – Provisions Windows VMs with cloudbase-init.
- **Modules:**
  - `compute-base` – Generates unique ED25519 SSH key pairs per host and manages shared settings.
  - `vm-linux` / `vm-windows` / LXC modules – Define Proxmox resources, cloud-init / cloudbase-init configuration, networking, and disks.
  - `ansible-inventory` – Produces Ansible inventory artifacts consumed by the `ansible/` project.
- **Outputs:**
  - Per-project inventories written into `ansible/inventory/` (e.g., `linux-vms-hosts.yml`).
  - SSH connection hints and `ssh_commands` outputs.

Each project is independent: it has its own state files under `generated/state/`, its own `terraform.tfvars.secret`, and its own `instances/*.auto.tfvars` describing hosts.

### Phase 2: Ansible – Unified Configuration

The `ansible/` project merges inventories from all Terraform projects into a single view and provides playbooks and roles for both Proxmox hosts and the K3s cluster.

- **Inventories (generated):**
  - `ansible/inventory/all-hosts.yml` – Merged inventory across all platforms.
  - `ansible/inventory/lxc-hosts.yml`, `linux-vms-hosts.yml`, `windows-vms-hosts.yml` – Per-platform subsets.
  - `ansible/inventory/k3s-hosts.ini` – K3s-specific nodes.
- **Roles:**
  - `roles/common_system` – Base system packages and OS updates.
  - `roles/common_users` – Non-root user creation and SSH key propagation.
  - `roles/common_ssh` – SSH hardening (disables root login, enforces key auth).
  - `roles/common_docker` / `common_podman` – Container runtime installation and `/srv/docker` layout.
  - `roles/k3s_cluster` / `k3s_system_prep` – K3s cluster management and node preparation.
- **Playbooks:**
  - `playbooks/lxc/bootstrap.yml`, `configure.yml` – Bootstrap and configure LXC/Linux VM hosts.
  - `playbooks/k3s/*.yml` – K3s cluster lifecycle and operations.

The execution order of roles is important (system → users → ssh → docker); running SSH hardening before user creation can lock you out. Always prefer the existing playbooks and `make` targets rather than ad hoc commands.

### Phase 3: GitLab CI/CD – Service Deployments

The GitLab pipeline (`.gitlab-ci.yml`) is split into a parent pipeline and dynamic child pipelines:

- **Parent pipeline stages:** `build-tools → test → lint → build-images → generate → trigger → security → publish → renovate`.
- **Child pipeline stages (per service):** `validate → preflight → backup → deploy → verify`.

Key characteristics:

- A "golden" CI image is built under `docker/ci/` and used for most jobs.
- Python orchestration in `scripts/ci/` manages service detection, pipeline generation, linting, and portfolio publishing.
- Bash scripts in `scripts/ci/` handle remote validation, backups, deployments, and health checks.
- Deployments are idempotent via checksum comparisons (`.checksum` files in `/srv/docker/<service>/`).
- A resource group prevents concurrent deployments to the same target.

On target hosts, services reside in `/srv/docker/<service>/` owned by `maintainer:docker`, with backups in `/srv/docker-backups/`.

## SSH Keys, Secrets & Naming Conventions

### SSH Keys

- Terraform generates unique ED25519 SSH key pairs per host during provisioning.
- Keys are stored on the control machine as:
  - `~/.ssh/<hostname>_id_ed25519` (private)
  - `~/.ssh/<hostname>_id_ed25519.pub` (public)
- Ansible bootstrap uses the Terraform-generated key to log in as `root`, then copies it to the non-root user and disables root SSH.

### Secrets & Sensitive Configuration

Secrets are intentionally spread across three systems; do not simplify this into a single location when modifying the repo.

- **Terraform:**
  - `terraform/<project>/terraform.tfvars.secret` – Proxmox credentials, backend tokens, and defaults (gitignored).
- **Ansible:**
  - `ansible/group_vars/all/vault.yml` – Encrypted with Ansible Vault. Use the existing `make vault-*` targets instead of editing directly.
- **GitLab CI/CD:**
  - SSH keys and host IPs: `SSH_KEY_<HOST>`, `HOST_<HOST>` (e.g., `SSH_KEY_PIHOLE_1`, `HOST_PIHOLE_1`).
  - Service environment variables: `ENV_<SERVICE>` (e.g., `ENV_PIHOLE_1`).
  - Name transformation: `hostname` like `pihole-1` → `PIHOLE_1` (hyphens to underscores, uppercased).

Do not introduce plaintext secrets into the repo; follow existing patterns and helper targets.

## Assistant Behavior & Existing AI Rules

This repository already defines AI assistant rules for Claude, Cursor, and Copilot. Warp should respect the same intent:

- **Do not create extra Markdown reports/files** (SUMMARY/AUDIT/etc.) unless the user explicitly requests them. Use chat responses instead.
- Treat tasks as a **Senior DevOps engineer**: favor Terraform/Ansible/CI automation, infrastructure-as-code, and minimal manual steps.
- Follow the established naming conventions from existing rules:
  - camelCase for variables/functions, PascalCase for classes, snake_case for filenames/directories, and UPPER_CASE for environment variables.
- Prefer modifying the existing workflows (Terraform Makefiles, Ansible Makefile, CI scripts) rather than introducing parallel, ad hoc tooling.
- Preserve the **single-host philosophy** for Ansible operations (use `*-host` targets with explicit `HOST=`) and the three-phase workflow boundaries (Terraform → Ansible → CI/CD).
